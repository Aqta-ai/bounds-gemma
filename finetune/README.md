# bounds-gemma · fine-tune

LoRA fine-tune of Gemma 4 E2B on a healthcare PHI corpus, packaged for
WebLLM (browser via MediaPipe) and Ollama (local daemon).

This directory is the training pipeline. The output is the published
model at `huggingface.co/Aqta-ai/bounds-gemma-e2b-phi-ft` (when ready).

## Step 1 — Generate the training corpus (local, ~3-4 hours)

Synthesises clinical-note paragraphs + labeled PHI spans using your
local Ollama daemon. Teacher: `gemma4:latest` (the 9.6 GB e4b variant).
Student target: `gemma4:e2b` (the 7.2 GB e2b variant).

Prerequisites:

```bash
ollama pull gemma4:latest   # ~9.6 GB
ollama serve                # daemon listening on localhost:11434 (or 11435 if 11434 is taken)
```

Run the generator:

```bash
OLLAMA_HOST=http://127.0.0.1:11434 EXAMPLES_TARGET=500 \
  node finetune/scripts/generate_phi_dataset.mjs
```

Output appears at `finetune/data/phi_training.jsonl`. Each line is
one OpenAI-style training example:

```json
{
  "messages": [
    { "role": "system", "content": "<production system prompt>" },
    { "role": "user", "content": "<clinical-note paragraph>" },
    { "role": "assistant", "content": "<JSON detections array>" }
  ]
}
```

The script supports resume: re-running picks up where the previous run
stopped (based on the line count of the existing JSONL). Adjust
`EXAMPLES_TARGET` to grow the corpus.

Quality controls baked into the generator:

- **Whitelisted categories.** Only `gemma:inline_diagnosis`,
  `gemma:medication_mention`, `gemma:treatment_procedure`,
  `gemma:indirect_health_context`, `gemma:sensitive_social`,
  `gemma:genetic_reference` survive validation. The teacher
  occasionally invents new categories (e.g. `gemma:symptom_mention`);
  those are silently dropped.
- **In-corpus check.** Every detection's `text` field must be a
  byte-identical (NFC-normalised) substring of the synthetic note.
  Paraphrases and hallucinations are dropped.
- **Confidence floor.** Detections below 0.75 confidence are removed,
  matching the production `HEALTHCARE_CONFIDENCE_FLOOR` constant.
- **Negative examples.** ~7% of generated examples use an empty
  category list and must produce zero detections; the validator
  accepts these to train the model when there's nothing to flag.

## Step 2 — Train (Colab T4, ~60-90 minutes)

Open `notebooks/train_in_colab.ipynb` in Google Colab. Set runtime to
T4 GPU. Upload `data/phi_training.jsonl` to the Colab session under
`finetune/data/`.

Run the notebook top to bottom. The training script does:

1. Load `unsloth/gemma-4-E2B-it-bnb-4bit` (4-bit quantised base)
2. Attach LoRA adapters (`r=16`, `alpha=16`, dropout=0)
3. Format the JSONL through Gemma's chat template
4. Train for 3 epochs at batch size 2 × grad-accum 4, LR=2e-4
5. Save the LoRA adapter
6. Merge into the base model and save as fp16
7. Convert to GGUF q4_k_m (for Ollama)

Outputs in the Colab workspace:

- `bounds-gemma-e2b-phi-ft/` — LoRA adapter (~50 MB)
- `bounds-gemma-e2b-phi-ft/merged/` — Full model (~5 GB fp16)
- `bounds-gemma-e2b-phi-ft-gguf/` — Ollama-ready GGUF

A smoke-test cell at the bottom runs the merged model on a sample
clinical note and reports parsed detections.

---

## Step 3 — Export to LiteRT-LM for WebLLM (Colab T4, ~15-20 minutes)

The browser path (MediaPipe LLM Inference via `@mlc-ai/web-llm`)
requires the model in LiteRT-LM `.task` format. The Colab notebook
runs `scripts/export_litert.py` to produce two variants:

- `bounds-gemma-e2b-phi-ft.task` — full kv-cache (`2048` tokens), ~2 GB
- `bounds-gemma-e2b-phi-ft-web.task` — slim kv-cache (`1024` tokens), ~1.5 GB

The web variant is what bounds.aqta.ai loads at runtime.

---

## Step 4 — Push to Hugging Face

The Colab notebook's last cell pushes the merged model + .task files
to `huggingface.co/Aqta-ai/bounds-gemma-e2b-phi-ft` (public) and the
GGUF to `huggingface.co/Aqta-ai/bounds-gemma-e2b-phi-ft-gguf`.

Requires a Hugging Face token with write access to the `Aqta-ai/*`
organisation, set in Colab Secrets as `HF_TOKEN`.

---

## Step 5 — Wire the fine-tune into bounds-gemma

Once the HF repo is public, update bounds-gemma's default model URL.
In `src/workers/gemma.worker.ts`, the WEBLLM_MODEL_ID constant
currently points at stock `gemma-4-E2B-it-q4f16_1-MLC`. Replace with
the bounds fine-tune:

```ts
const WEBLLM_MODEL_ID = process.env.GEMMA_MODEL_URL ||
  'https://huggingface.co/Aqta-ai/bounds-gemma-e2b-phi-ft/resolve/main/bounds-gemma-e2b-phi-ft-web.task'
```

And for the Ollama path, document the new local pull:

```bash
ollama pull Aqta-ai/bounds-gemma-e2b-phi-ft-gguf:latest
```

## Honest scope and known limitations

This is a **distillation fine-tune from synthetic data**, not a
clinically-validated medical-grade model. The training corpus is
generated by Gemma 4 E2B teacher itself, which means:

- The model learns to imitate the teacher's labelling decisions, NOT
  the ground truth of human-annotated clinical notes
- Adversarial paraphrases, rare conditions, and non-English notes are
  underrepresented in the corpus and will produce more false
  negatives
- A held-out evaluation against an i2b2 / n2c2 de-identification
  corpus (requires DUA clearance) is **future work** and not part of
  this submission

The submission claim is therefore **"fine-tuned for the contextual
PHI shapes Bounds catches"**, not "production-grade clinical
redaction." The full safety properties of the toolkit (in-corpus
verification, confidence floor, default-off in the review UI) all
apply regardless of which model serves the spans.

## Special Tech Track eligibility

This fine-tune uses **Unsloth** for the LoRA training (see
`scripts/train.py`). That makes the submission eligible for the
Gemma 4 Good *Unsloth Special Tech Prize*: *"For the best fine-tuned
Gemma 4 model created using Unsloth, optimized for a specific,
impactful task."*

The impactful task: contextual PHI redaction for HIPAA Safe Harbor
identifier #17 ("any other unique identifying number, characteristic,
or code"), the catch-all category that regex and NER demonstrably
miss.
